大语言模型并不是直接读取文字。

在模型眼里，文本会先被 tokenizer 切成 token，再转换成数字 ID。模型真正处理的是这些 ID 对应的向量，而不是我们看到的汉字、英文单词或标点。

## token 是什么

token 可以是一个词，也可以是词的一部分，还可以是一个汉字、标点或空格片段。

例如英文句子：

```text
unbelievable
```

可能被切成：

```text
un + believable
```

也可能被切成更细的子词。具体怎么切，取决于 tokenizer 的训练方式和词表。

中文里，一个 token 可能对应一个汉字，也可能对应多个汉字组成的常见词。不同模型的 tokenizer 不一样，所以同一句话在不同模型里消耗的 token 数也不一样。

## 为什么不按字切分

最简单的办法是按字符切分，但这样序列会变长，模型计算成本会变高。

如果按词切分，又会遇到另一个问题：世界上总会出现新词、拼写变化、代码符号、URL、表情、缩写。完全基于词的词表很难覆盖所有情况。

子词切分是折中方案。它让常见词保持较短表示，同时也能把罕见词拆成可处理的小片段。

## BPE 的直觉

BPE 是常见 tokenizer 方法之一。

它的核心思想很朴素：从字符开始，反复合并最常一起出现的相邻片段。

简化过程如下：

```text
l o w
l o w e r
n e w e r

合并高频片段：
lo, low, er ...
```

训练结束后，模型得到一个词表。推理时，tokenizer 会按照这个词表把新文本切成 token。

## tokenizer 会影响什么

第一，影响上下文长度。

模型常说支持 8k、32k、128k context，这里的长度单位是 token，不是字符。中文、英文、代码、表格的 token 消耗方式都不一样。

第二，影响多语言表现。

如果 tokenizer 对某种语言切得特别碎，那么同样的信息会占用更多 token，也会增加模型学习难度。

第三，影响代码和结构化文本。

代码里有大量符号、缩进和特殊命名。如果 tokenizer 对代码不友好，模型处理代码时就会更吃力。

## 特殊 token

除了普通文本 token，模型还会使用特殊 token。

常见例子：

```text
<bos>  序列开始
<eos>  序列结束
<pad>  填充
<unk>  未知 token
```

聊天模型还会有表示 system、user、assistant 的特殊模板或控制 token。它们帮助模型区分消息角色和对话结构。

## 一个常见误区

很多人把 token 当成“字数”，这是不准确的。

同样 100 个汉字、100 个英文单词、100 行代码，token 数可能差异很大。估算成本、上下文长度和输入限制时，都应该以 tokenizer 的实际结果为准。

## 总结

Tokenizer 是文本进入大模型之前的第一道门。

它决定文本怎么被切分，决定输入会消耗多少 token，也影响模型处理不同语言和代码的效率。理解 tokenizer，能帮助你更准确地理解上下文窗口、调用成本和提示词长度限制。
