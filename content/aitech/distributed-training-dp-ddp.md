## 数据并行解决什么问题

数据并行的核心思路很直接：模型复制多份，数据拆成多份，每张 GPU 处理自己的 mini-batch 分片，然后同步梯度。

假设有 4 张 GPU，一个 batch 可以切成 4 份：

- GPU 0 处理第 1 份数据。
- GPU 1 处理第 2 份数据。
- GPU 2 处理第 3 份数据。
- GPU 3 处理第 4 份数据。

每张卡都做一次前向和反向传播，得到自己的梯度。随后所有 GPU 通过通信操作把梯度同步成一致的结果，再各自更新本地模型参数。

## DP：简单但不够高效

PyTorch 里的 `torch.nn.DataParallel` 是最容易上手的数据并行方式。它会在每张 GPU 上复制模型，把输入数据切分后分发到不同 GPU，最后再把结果聚合回来。

它的优点是简单，通常只需要在模型外面包一层：

```python
model = torch.nn.DataParallel(model)
```

但 DP 的缺点也很明显：

- 单进程多线程，调度和通信效率有限。
- 主卡需要负责数据分发和结果聚合，容易成为瓶颈。
- 每张卡仍然保存完整模型，不能解决单卡放不下模型的问题。

所以 DP 更适合小模型、临时实验，或者只想快速确认多卡能跑通的场景。

## DDP：多卡训练的默认选择

`DistributedDataParallel`，也就是 DDP，是 PyTorch 官方推荐的多 GPU 训练方式。

DDP 通常采用“单进程单卡”的模式：每个进程绑定一张 GPU，每个进程里都有一份完整模型。训练时，各进程独立读取不同数据分片，完成前向和反向传播；反向传播过程中，DDP 会通过 NCCL 等通信后端同步梯度。

一个典型 DDP 训练循环大致是：

```python
for batch in dataloader:
    optimizer.zero_grad()
    loss = model(batch)
    loss.backward()
    optimizer.step()
```

代码看起来和单卡训练非常接近，区别主要在初始化进程组、设置 sampler、绑定 GPU 这些外围步骤。

## DDP 为什么更快

DDP 的关键优化在于梯度同步发生在反向传播过程中。模型的某一层梯度一旦计算完成，就可以开始通信，而不是等所有层的反向传播全部结束后再统一同步。

这让计算和通信有机会重叠，从而减少等待时间。

相比 DP，DDP 还有几个工程优势：

- 每张 GPU 对应一个独立进程，避免 Python GIL 和单进程调度瓶颈。
- 通信路径更直接，主卡不会承担过多聚合压力。
- 和分布式采样、混合精度、梯度累积等训练技巧配合更自然。

## DDP 仍然解决不了什么

DDP 的前提是每张 GPU 都能放下一份完整模型。

这意味着模型参数、梯度、优化器状态都要完整存在于每张卡上。如果模型本身已经超过单卡显存，DDP 就无能为力。哪怕模型勉强能放下，也可能因为 batch size 太小导致训练吞吐不理想。

这时就需要 FSDP 或 ZeRO 这类更进一步的分片训练方案。

## 实战建议

如果你刚开始做多卡训练，可以按这个顺序走：

1. 单卡先跑通训练逻辑。
2. 用 DDP 扩展到多卡，确认 loss、吞吐、保存加载都正常。
3. 加入混合精度、梯度累积、梯度裁剪等常规训练技巧。
4. 如果显存依然不够，再考虑 FSDP、DeepSpeed ZeRO 或模型并行。

DDP 不一定是最大模型训练的终点，但它是理解分布式训练最稳的起点。
