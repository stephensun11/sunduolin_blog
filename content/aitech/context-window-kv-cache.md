使用大模型时，经常会看到两个词：上下文窗口和 KV Cache。

它们都和推理成本有关。理解它们，可以帮助你更好地设计提示词、控制成本，也能解释为什么长对话会越来越慢、越来越贵。

## 上下文窗口是什么

上下文窗口指模型一次能处理的 token 数量。

如果一个模型支持 32k context，意思是输入、历史对话、系统提示和模型即将生成的内容加起来，最多大约 32k token。

这里的单位是 token，不是字数。

```text
系统提示 + 历史消息 + 当前问题 + 输出内容 <= 上下文窗口
```

超过窗口后，要么报错，要么需要截断旧内容，要么使用摘要、检索等方式压缩上下文。

## 长上下文不等于全部有效

模型支持很长上下文，并不代表它能完美使用所有上下文。

长文本里可能出现信息稀释、位置偏差和注意力分散。模型有时更关注开头和结尾，中间信息反而容易被忽略。

所以长上下文不是万能药。很多时候，把相关信息检索出来放进短上下文，比把所有材料都塞进去更有效。

## KV Cache 是什么

Decoder-only 模型生成文本时，每次只生成一个新 token。

如果每生成一个 token 都重新计算整个上下文，成本会非常高。KV Cache 的作用就是缓存历史 token 的 Key 和 Value。

这样生成下一个 token 时，模型只需要计算新 token 的 Q/K/V，并复用之前缓存的 K/V。

```text
历史 token 的 K/V -> 缓存
新 token -> 只计算新增部分
```

这能显著加速自回归生成。

## KV Cache 为什么占显存

KV Cache 需要保存每一层、每个 attention head、每个历史 token 的 K/V 向量。

上下文越长，batch 越大，模型层数越多，KV Cache 占用就越高。

这就是长对话推理变贵的原因之一：不仅注意力计算变多，缓存也会越来越大。

## 实际使用建议

第一，不要无脑保留完整历史对话。

可以定期摘要旧消息，只保留当前任务真正需要的信息。

第二，把稳定规则放在 system prompt，把临时材料放在当前用户消息里。

第三，对于长文档问答，优先考虑检索相关段落，而不是整篇塞给模型。

第四，写提示词时尽量减少重复背景，尤其是多轮调用中的重复说明。

## 总结

上下文窗口决定模型一次能看多少 token，KV Cache 决定生成时如何复用历史计算。

长上下文带来了更强的输入能力，也带来了更高的计算和显存成本。真正高效的 LLM 应用，不是把所有东西都塞进上下文，而是让模型只看到当前最需要的信息。
