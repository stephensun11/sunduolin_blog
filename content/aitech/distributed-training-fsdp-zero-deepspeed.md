## 显存到底花在哪里

大模型训练时，显存通常不只被模型参数占用。完整训练还需要保存：

- 模型参数。
- 反向传播产生的梯度。
- 优化器状态，例如 Adam 的一阶动量和二阶动量。
- 激活值，也就是前向传播中为了反向传播保留下来的中间结果。

DDP 会在每张 GPU 上保留一整套模型参数、梯度和优化器状态。卡越多，吞吐可能越高，但这些状态本身并没有被拆开，所以显存瓶颈仍然存在。

FSDP 和 ZeRO 的核心问题就是：这些状态能不能不要每张卡都存一份？

## ZeRO 的核心思想

ZeRO 是 Zero Redundancy Optimizer 的缩写，目标是减少数据并行中的冗余存储。

它通常分为三个阶段：

| 阶段 | 分片对象 | 直观效果 |
| --- | --- | --- |
| ZeRO-1 | 优化器状态 | 减少 Adam 等优化器带来的额外显存 |
| ZeRO-2 | 优化器状态 + 梯度 | 进一步减少反向传播后的梯度占用 |
| ZeRO-3 | 优化器状态 + 梯度 + 参数 | 连模型参数本身也分片保存 |

ZeRO-3 最彻底：每张 GPU 只保存模型状态的一部分。需要计算某一层时，再把这层所需参数临时聚合起来；计算完成后，又释放或重新分片。

这种方式能显著降低单卡显存压力，但代价是通信变多，调试复杂度也上升。

## FSDP 做了什么

FSDP 全称是 Fully Sharded Data Parallel。它把模型参数、梯度和优化器状态都切分到不同 GPU 上。

训练过程可以简化成四步：

1. 每张 GPU 平时只保存自己负责的参数分片。
2. 前向传播需要某一层时，通过通信临时 gather 出完整参数。
3. 计算完成后释放完整参数，只保留分片。
4. 反向传播和优化器更新时，也按分片方式同步梯度和状态。

如果有 N 张 GPU，理想情况下，参数、梯度和优化器状态的单卡存储都可以接近降到原来的 `1/N`。

## FSDP-v1 与 FSDP-v2

很多资料会把 FSDP 分成 v1 和 v2 来讲。对入门者来说，不需要一开始陷入 API 细节，先抓住共同点：它们都是围绕“全分片”这件事展开。

FSDP-v1 更像传统的模块包裹式使用方式，常见写法是把模型或部分子模块包进 `FSDP(...)`。

FSDP-v2 更强调和 PyTorch 新一代分布式抽象配合，让张量分布、设备网格、并行组合等概念更统一。你可以把它理解为工程接口和组合方式的升级，核心目标仍然是降低大模型训练时的显存冗余。

## DeepSpeed 和 FSDP 怎么选

DeepSpeed 是一个更完整的分布式训练框架。它不只是实现 ZeRO，还集成了 offload、pipeline parallel、tensor parallel、混合精度、checkpoint 等能力。

FSDP 的优势是 PyTorch 原生，和普通 PyTorch 项目结合比较自然。如果你的训练代码主要在 PyTorch 生态内，且需求集中在全分片数据并行，FSDP 是很值得优先考虑的选择。

DeepSpeed 的优势是训练大模型的工程能力更全。如果你需要 ZeRO-Offload、复杂并行策略、成熟的大模型训练配置体系，DeepSpeed 往往更省事。

一个简单判断：

- 已有 PyTorch 训练代码，想尽量少引入框架：先看 FSDP。
- 要训练非常大的模型，并且需要 offload、pipeline、复杂配置：优先看 DeepSpeed。
- 只是单机多卡中等模型：DDP 可能已经足够。

## 代价与坑

分片训练不是免费午餐。它节省显存，但会增加通信和状态切换。

常见问题包括：

- 通信量上升，网络或 NVLink 带宽成为瓶颈。
- checkpoint 保存和恢复更复杂。
- 不同模块的 wrap 策略会影响性能。
- 小模型使用 FSDP/ZeRO 可能收益不明显，甚至更慢。

所以实践里不要一开始就把所有复杂工具都打开。先用 DDP 建立基线，再逐步引入 FSDP 或 ZeRO，会更容易判断每一步带来的收益和问题。

## 小结

DDP 的问题是“每张卡都存完整状态”。ZeRO 和 FSDP 的答案是“把状态拆开存”。DeepSpeed 则是在 ZeRO 思想上构建出更完整的大模型训练框架。

理解了这一层，很多大模型训练配置就不会只是神秘参数，而是围绕显存、通信和工程复杂度做取舍。
