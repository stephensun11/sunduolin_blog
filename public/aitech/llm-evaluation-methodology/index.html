LLM 评测方法论：从离线基准到线上质量闭环

2026-07-12 AiTech LLM 评测 LLM Evaluation Benchmark LLM Judge 数据污染 线上评测

大模型评测最危险的状态不是“没有分数”，而是拥有一个看起来精确、实际上回答了错误问题的分数。

模型版本、提示模板、采样参数、解析器、数据污染、评审模型和服务配置都能改变结果。严谨评测的目标不是寻找唯一排行榜，而是建立一套能支持决策、能复现、能解释失败的证据链。

先写评测契约

在运行任何 benchmark 前，先回答：

  1. 要支持什么决策：选底座、验收微调、发布回归还是容量规划？
  2. 目标用户和请求分布是什么？
  3. 哪些错误代价最高？
  4. 模型、系统和数据各由什么指标负责？
  5. 多大的变化才算具有实际意义？

如果目标是客服机器人，MMLU 提升不一定抵消错误退款的风险；如果目标是代码代理，单轮代码补全分数也不能代替仓库级任务成功率。

评测矩阵：不要只测能力

HELM 强调多场景、多指标评估。一个完整矩阵至少有四个轴：

轴 典型内容
场景 问答、代码、数学、检索、多轮、工具调用
质量 正确性、相关性、完整性、事实性、校准
风险 安全、偏见、隐私、鲁棒性、过度拒绝
系统 TTFT、TPOT、吞吐、可用性、成本

最终可以有总览，但原始分域结果必须保留。把所有指标压成单个加权分数会隐藏取舍，权重本身也是产品决策。

静态 benchmark 能告诉你什么

MMLU 覆盖 57 个学科，用多选题衡量知识与问题求解；MMLU-Pro 增加难度、扩展选项并提高对提示变化的稳定性；LiveBench 通过持续更新题目降低污染并保持挑战性。

它们适合标准化比较，但结论有边界：

  • 多选准确率不等于开放式回答质量；
  • 学科覆盖不等于真实用户分布；
  • benchmark 可能饱和或被训练数据污染；
  • prompt 和 few-shot 示例会改变分数；
  • 排行榜差异可能小于统计不确定性。

正确的表述是“模型在该版本、该提示和该评分协议上的结果”，而不是“模型智力为 82 分”。

可复现协议包含什么

每次运行应冻结：

model_revision: exact-commit-or-checksum
tokenizer_revision: exact-commit-or-checksum
chat_template: versioned-template
benchmark_revision: dataset-commit
evaluation_code: git-commit
fewshot: 5
seed: 20260712
temperature: 0
top_p: 1
max_tokens: 2048
stop: [...]
dtype: bf16
engine: vllm-x.y.z
hardware: 8xH100-80GB

对闭源 API，还要记录模型快照名、运行日期、区域、重试和速率限制。供应方可能更新同名模型，时间是复现信息的一部分。

lm-evaluation-harness 等框架可以统一任务和请求格式，但使用框架不自动保证公平。仍需检查 task 版本、模板、答案解析和模型后端实现。

生成设置必须与任务一致

确定性知识评测

常使用 greedy 或 temperature=0，减少采样方差。但某些 API 的并行 kernel 仍可能产生微小非确定性。

创意或多解任务

只采样一次不足以描述分布。应固定 temperature/top-p，使用多个 seed，并报告均值、方差或成功率。

推理任务

需要明确是否允许 Chain-of-Thought、最大输出长度、工具和自洽采样。允许更多推理 token 通常会增加成本，比较质量时也要报告计算预算。

评分器必须匹配输出空间

Exact Match

适合唯一短答案，但应先定义规范化：大小写、空格、标点、单位、分数和等价表达。规范化规则必须固定，不能看到模型输出后临时修改。

F1、ROUGE、BLEU

它们衡量词或 n-gram 重合，在抽取和特定文本生成中有用，但可能低估语义等价答案，也可能奖励复制而非事实正确。

可执行评分

代码单元测试、SQL 执行、数学符号验证通常比字符串重合更可靠，但测试覆盖决定可信度。执行环境必须沙箱化并限制网络、文件和资源。

pass@k

代码生成常从 (n) 个样本中观察到 (c) 个正确样本，pass@k 的无偏估计常写为：

\[ \operatorname{pass@k}=1-\frac{\binom{n-c}{k}}{\binom{n}{k}} \]

它衡量 k 次尝试至少一次成功的概率，不等同于单次准确率。比较时必须对齐 n、k 和采样参数。

LLM-as-a-Judge 怎样使用

开放式写作、对话和摘要难以用规则评分，常让强模型做评审。它扩展快，但不是客观真值。

常见偏差包括：

  • 位置偏差：偏爱 A 或第一项；
  • 长度偏差：更长看起来更完整；
  • 自家模型偏差：偏爱相似风格；
  • verbosity 与格式捷径；
  • 对错误引用或复杂推理缺乏核验；
  • prompt injection：候选回答试图操纵评审。

降低风险的方法：

  1. 采用明确 rubric，逐维打分；
  2. 交换 A/B 顺序并检查一致性；
  3. 隐藏模型身份；
  4. 对评审输出使用结构化 schema；
  5. 用人工标注集校准 judge 与人类的一致率；
  6. 对高风险事实调用检索、执行器或专家复核；
  7. 保存 judge 模型版本、prompt 和原始理由。

judge 分数应被视为一个测量工具的输出，而不是事实本身。

Pairwise 比绝对打分更可靠吗

对于难以校准的开放回答，比较 A 与 B 往往比独立打 1-10 分稳定。可统计胜、负、平，并用 Bradley-Terry 等模型估计相对强度。

但 pairwise 也有局限：

  • 只提供相对结论，不说明二者是否都不可用；
  • 顺序和长度偏差仍存在；
  • 多模型全量两两比较成本高；
  • 不同任务上的偏好可能不可传递。

因此 pairwise 最好与绝对质量阈值和失败标签结合。

统计不确定性不能省略

测试集是任务分布的有限样本。准确率 (\hat p) 不是无误差真值。

对于复杂指标，bootstrap 是实用方法：重复从题目中有放回采样，计算指标分布，再报告置信区间。比较同一批题上的两个模型时，应做 paired bootstrap，保留题目配对，比独立抽样更有统计效率。

同时区分：

  • 统计显著：差异不太可能由有限样本随机性造成；
  • 实际显著：差异足以改变产品决策。

十万题上的 0.1% 可能统计显著，却没有业务价值；一项高风险能力的 2% 回退即使样本少，也可能阻止发布。

数据污染怎样检查

污染包括训练集直接包含题目、答案、题解、翻译或派生版本。它会让分数高估泛化。

检查方法包括：

  • 精确和规范化字符串匹配；
  • n-gram 或 MinHash 近似匹配；
  • 检索题目在网页、代码仓库和数据快照中的副本；
  • 检查模型对随机扰动题目的敏感性；
  • 使用持续更新或私有评测集；
  • 报告 cutoff 与 benchmark 发布时间。

不存在一个仅靠模型输出就能完全证明“未污染”的测试。最强证据来自可审计训练数据和 benchmark 去污染流程。

领域评测应该怎样构造

真实产品集通常比公开 benchmark 更重要。构建步骤：

  1. 从真实请求中分层采样并脱敏；
  2. 建立任务 taxonomy 与风险等级；
  3. 由领域专家编写理想答案或评分 rubric；
  4. 保留简单、典型、长尾和对抗样本；
  5. 划分开发集、回归集和严格保密测试集；
  6. 定期新增时间敏感案例，但不要覆盖历史版本；
  7. 对标注分歧进行复审并记录一致率。

测试集不应只包含历史失败，否则会夸大极端案例并失去分布代表性。应同时有代表性集和 challenge set。

RAG 与 Agent 要做组件级评测

端到端失败可能来自多个环节。

RAG

  • 检索 recall@k：支持答案的文档是否被召回；
  • 排序指标：相关文档是否靠前；
  • context precision：送入模型的内容有多少相关；
  • faithfulness：回答是否由上下文支持；
  • answer correctness：最终答案是否正确；
  • citation correctness：引用是否准确指向证据。

Agent

  • 任务成功率；
  • 工具选择与参数正确率；
  • 每任务步数、token、延迟和成本；
  • 恢复率与不可逆错误率；
  • 权限越界和提示注入防护；
  • 在环境变化下的鲁棒性。

只有端到端分数，无法指导修复；只有组件分数，又无法保证用户任务真正完成。两层都要保留。

系统性能怎样测

质量相同的模型，服务体验仍可能不同。常用指标：

  • TTFT（Time to First Token）：请求到首 token；
  • TPOT（Time per Output Token）：首 token 后每个输出 token 的平均时间；
  • ITL（Inter-token Latency）：相邻 token 间延迟分布；
  • 请求吞吐与 token 吞吐；
  • P50/P95/P99 延迟；
  • 峰值显存、GPU 利用率和功耗；
  • 错误率、超时率和队列时间。

性能结果必须附带输入长度、输出长度、并发、batch 策略、硬件和精度。只报告“tokens/s”而不说明是单请求还是全服务器吞吐，几乎没有可比性。

线上评测：真实分布与因果问题

A/B 测试可以观察任务完成率、用户修正率、会话中断、延迟和成本。但用户反馈不是纯质量标签：界面、等待时间、用户群和曝光策略都会影响行为。

上线实验应：

  • 随机分流并固定用户归组；
  • 预先定义主指标和护栏指标；
  • 控制模型外的产品变化；
  • 监控样本比例不匹配；
  • 为高风险任务设置人工审核或渐进放量；
  • 保留回滚条件；
  • 避免反复偷看结果后提前停止造成假阳性。

线上胜出也不能替代安全评测，因为低频高损事件很难在短期 A/B 中充分出现。

回归门禁怎样设计

一个发布门禁可以分层：

L0: 构建、加载、确定性 smoke test
L1: 快速核心能力和格式回归
L2: 完整离线能力/安全/领域评测
L3: 性能与容量压测
L4: 小流量 shadow 或 canary
L5: A/B 与持续监控

每层设置硬门槛和允许波动。平均提升不能自动抵消关键安全项下降。失败样本应进入带版本的错误库，而不是直接回流训练集。

评测报告应包含什么

  • 目的、目标人群和决策阈值；
  • 模型、数据、代码、模板和推理配置版本；
  • 数据集来源、规模、时间范围和污染检查；
  • 分域结果、宏/微平均与置信区间；
  • 与基线的 paired 差异；
  • 质量、风险、性能与成本；
  • judge 校准与人工一致率；
  • 失败案例 taxonomy；
  • 已知盲区和不可比较项；
  • 可重放的原始输出与运行清单。

常见误区

只报告最高的一次结果

这会隐藏 seed、采样和提示方差。应预先固定协议并报告重复运行。

prompt 为每个模型单独调到最好再直接比较

这回答的是“每个模型在各自最佳系统中的上限”，不是“同一接口下谁更好”。两种比较都可以，但必须明确。

用公开榜单替代产品验收

公开 benchmark 提供公共坐标，不包含你的用户、工具、知识库和风险分布。

judge 与被测模型来自同一版本族就一定可靠

相似训练数据和风格可能造成系统性偏好，需要人工校准与多评审交叉检查。

没有显著下降就算通过

样本太少时，“未发现差异”不等于“证明等价”。要报告检测能力和置信区间。

总结

LLM 评测是一套测量系统，不是一张排行榜。它要把问题定义、样本、推理配置、评分器、统计方法和发布决策连接起来。

最可信的结论通常不是“新模型全面更强”，而是更具体的陈述：它在哪些用户场景、以多少成本、在怎样的不确定性和风险边界下改善了什么。只有这种结论，才足以指导训练和上线。
