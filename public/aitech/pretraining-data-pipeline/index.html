LLM 预训练数据工程：从原始语料到可复现的数据配方

2026-07-12 AiTech LLM训练 LLM 预训练数据 数据清洗 数据去重 数据配比 数据污染

大模型训练通常被描述成“给 Transformer 喂很多 token”。这句话没有错，却容易掩盖真正困难的部分：token 并不是天然存在的，它们来自一条需要反复验证的数据流水线。

模型最终学到什么，不只由参数量和算力决定。语料的来源、重复率、时间范围、语言比例、许可证、质量过滤和采样方式，都会改变模型的知识边界、表达风格、记忆风险和评测可信度。

本文讨论的是预训练数据工程，而不是 SFT 数据制作。两者都重视质量，但目标不同：预训练语料要覆盖广泛分布，SFT 数据则要精确示范期望行为。

一条完整的数据流水线

典型流程可以写成：

数据源登记
  -> 抓取与快照
  -> 解析和正文抽取
  -> 语言与内容识别
  -> 规则过滤
  -> 质量评分
  -> 精确/近似去重
  -> 隐私与合规处理
  -> 评测集去污染
  -> 分域配比与采样
  -> Tokenize、切分和打包
  -> 数据清单、版本和审计报告

顺序不是随意的。例如先去掉明显无效页面，可以降低后续 MinHash 去重的计算量；先完成去污染再切成训练序列，才能保留文档级溯源信息。

第一步：定义数据契约

在下载语料之前，先定义每条文档必须携带的元数据：

  • source_id：数据源和快照版本；
  • document_id：稳定的文档标识；
  • url 或原始位置；
  • 抓取时间、语言和许可证；
  • 内容哈希与处理流水线版本；
  • 过滤原因、质量分数和去重簇编号。

这不是形式主义。没有稳定标识，就无法回答“某段文字为什么进入模型”“一个过滤规则影响了多少 token”“评测泄漏来自哪个快照”。

训练数据也不应被视为一个不断覆盖的目录。更可靠的做法是使用不可变快照：原始层只追加，处理层由代码版本和配置生成，最终训练清单记录所有输入分片的校验和。

第二步：选择数据源

常见来源包括网页、书籍、论文、新闻、百科、论坛、代码、数学文本和人工整理数据。每类数据同时带来能力和偏差：

来源 主要价值 主要风险
网页 覆盖广、更新快 模板噪声、SEO 文本、重复和隐私信息
书籍 长篇连贯、编辑质量高 版权和年代偏差
论文 专业知识与论证结构 公式解析、引用噪声、领域分布偏斜
代码 编程与形式化能力 自动生成代码、许可证、密钥泄漏
论坛 对话和真实问题 毒性、低可信事实和个人信息
百科 稳定的事实性文本 风格单一、知识更新时间固定

“公开可访问”不等于“允许用于训练”。工程上需要记录许可证、使用条款、删除请求和地域合规要求；技术过滤无法替代法律审查。

第三步：解析与正文抽取

网页原始 HTML 里可能有导航、广告、Cookie 提示、相关推荐和隐藏文本。简单删除标签会把这些模板混入正文。

正文抽取至少要处理：

  • HTML 实体、Unicode 规范化和异常编码；
  • 标题、段落、列表、表格和代码块的结构；
  • 页眉、页脚、菜单和重复侧栏；
  • PDF 的断行、页码、双栏顺序和 OCR 错误；
  • 文档边界，避免把多个页面无声拼接。

不要过早把所有内容压成一行纯文本。结构信息以后可以用于质量判断、代码识别、段落级去重和长文档切分。

第四步：语言识别不是一个标签

语言分类器通常输出概率分布，而不是绝对真值。代码混合、中英混排、短文本和转写文本都可能误判。

更稳妥的策略是：

  1. 保存语言概率而非只保存 zh、en 标签；
  2. 对短文档使用更高阈值或单独规则；
  3. 保留“混合语言”和“未知”类别；
  4. 在文档级判断之外抽样检查段落级分布；
  5. 用人工标注集监控不同语言上的精确率和召回率。

语言配比要按 token 统计，而不是按文档数统计。一个英文长文档与一条中文短帖不应获得相同权重。

第五步：规则过滤与质量模型

规则过滤适合删除确定性较高的异常：正文过短、符号比例过高、重复行过多、占位文本、乱码、模板化页面或明显的机器生成垃圾。

质量模型则适合处理连续边界。可以用高质量参考语料训练分类器，或用语言模型困惑度、字符统计、结构特征共同评分。

但“像百科”不等于“高质量”。如果质量模型只学习某一种正式文体，它会系统性删除方言、口语、少数语言和真实社区讨论。质量过滤必须做分域消融：观察规则对每个来源、语言、主题和时间段删除了什么。

一个实用的验收表至少包括：

指标 目的
保留文档/token 比例 发现过滤突变
各语言与来源保留率 发现分布偏差
随机样本精确率 判断保留下来的文本是否可用
被删样本误杀率 判断过滤是否过强
下游小模型消融 判断过滤是否真的改善学习

Dolma 的数据报告展示了来源登记、过滤、去重、PII 处理和消融实验如何组合成可审计的语料配方。

第六步：精确去重与近似去重

去重至少分三层。

URL 和文档精确去重

对规范化 URL、原始内容哈希或规范化正文哈希进行去重，成本低，适合作为第一层。规范化不能过度：删除所有数字可能把不同年份的报告误判成同一文档。

段落级去重

网页模板、版权声明和转载尾注可能在不同正文中重复。段落级去重能删除这些高频片段，但要保留段落边界，避免删除后把前后不相关文本拼在一起。

近似去重

完全相同的哈希抓不到轻微改写、格式变化和局部转载。常见方法是把文档转成字符或 token 的 n-gram 集合，用 Jaccard 相似度衡量重合：

\[ J(A,B)=\frac{|A\cap B|}{|A\cup B|} \]

直接两两比较代价过高，因此常用 MinHash 近似 Jaccard，再用 LSH 找候选簇。阈值不是越低越好：技术文档共享 API 名称，新闻共享背景段落，代码共享许可证，错误阈值会误删有价值内容。

Lee 等人的研究 表明，去重不仅减少记忆式复现，也能降低训练步数并减少训练集与验证集重叠。需要注意：这是实证结果，不意味着任何去重配置都必然提升所有模型。

第七步：隐私、安全与合规处理

PII 检测通常包括邮箱、电话号码、IP、身份证件、地址、账号和密钥。规则方法精度高但覆盖有限，模型方法覆盖更广但成本和误报更高。

合理的策略是分级处理：

  • 对高置信度敏感字段替换或删除；
  • 对高密度敏感文档整篇剔除；
  • 对代码扫描 API key、私钥和凭据模式；
  • 保存删除规则版本，但不要在日志里再次写出敏感原文；
  • 建立可追溯的删除和重构机制。

有害内容过滤也要区分“描述有害行为”和“鼓励有害行为”。医学、新闻和安全研究文本可能包含敏感词，却仍有学习价值。

第八步：评测集去污染

如果评测题或答案出现在训练集里，分数可能反映记忆而非泛化。去污染应在训练前完成，并覆盖：

  • 题干的精确匹配和规范化匹配；
  • 题干与答案拼接后的 n-gram 匹配；
  • 近似复述和翻译版本；
  • benchmark 的网页镜像、题解和 GitHub 仓库；
  • 同一数据集的开发集、测试集及衍生版本。

污染判断要保存证据和阈值。命中短语不一定就是泄漏，例如常见数学定理会自然重复。更可靠的报告应同时给出污染前后分数，并说明删除单位是文档、段落还是窗口。

第九步：数据配比与采样

把所有来源直接拼接，相当于让体量最大的来源决定训练分布。实际训练通常为不同域设置采样权重。

设第 (i) 个数据域包含 (n_i) 个 token，一种温度采样写法是：

\[ p_i=\frac{n_i^{\alpha}}{\sum_j n_j^{\alpha}},\qquad 0<\alpha\le 1 \]

当 (\alpha=1) 时按原始体量采样；减小 (\alpha) 会提高小语种或小领域的相对权重。提高权重不等于凭空增加信息：小数据被重复过多会加重过拟合和记忆。

配比应通过小规模代理实验验证。比较相同参数量、token 预算和训练配置下的损失曲线及分域评测，才能把收益归因到数据，而不是算力差异。

第十步：Tokenize、切分和 packing

Tokenizer 必须在最终语料分布上验证。除了平均字符/token 比，还要检查不同语言、代码、数字、公式和特殊字符的压缩率。

训练样本通常有固定长度。常见做法有：

  • 连续拼接：文档间插入 EOS，再切固定窗口，token 利用率高；
  • 文档内切分：不跨文档，语义边界清晰但 padding 更多；
  • sequence packing：把多个短样本装入一个序列，并用 attention mask 隔离或用 EOS 明确边界。

如果简单拼接却没有 EOS，模型会把两个无关文档当成自然连续文本。若 packing 的 mask 实现错误，样本之间可能发生信息泄漏。

每个训练样本应能反查到原始文档和 token 区间。只有这样，异常梯度、记忆案例和删除请求才有机会定位。

数据规模与模型规模怎样配合

在固定训练 FLOPs 下，参数更多并不总是更好。Chinchilla scaling law 的核心结论是：在其研究范围和假设下，计算最优训练需要随参数量同步增加训练 token 数。

不要把论文结果误读成一个跨架构、跨数据质量永远固定的常数。真实配方还受推理成本、重复训练轮数、数据质量、模型结构和目标能力影响。它更重要的启示是：规划模型时必须同时规划数据和算力，不能只报参数量。

如何保证可复现

每次数据发布至少要冻结：

dataset_version: 2026-07-v1
source_snapshots: [...]       # 来源及日期
pipeline_commit: abc1234      # 处理代码版本
filter_config_hash: ...       # 阈值与规则
tokenizer_hash: ...
mixture_weights: {...}
shard_checksums: {...}
random_seed: 20260712

同时发布 datasheet：描述来源、用途、局限、许可证、人口与语言分布、过滤统计、已知偏差和联系方式。只记录“共 2T token”远远不够。

常见失败模式

只看最终 token 数

同样一万亿 token，可能是高质量多样文本，也可能是同一批页面反复转载。规模不能替代分布报告。

用困惑度作为唯一质量标准

低困惑度可能只是文本模板化或与评分模型训练域相似。应结合规则、人工抽检和下游消融。

去重后不重新计算配比

不同来源的重复率不同。去重会改变实际域分布，原先的采样权重需要重新计算。

把测试集文件删掉就算去污染

题解、镜像、复述和答案讨论仍可能存在。污染检测必须面向内容，而非只面向文件名。

无法从训练 shard 回溯来源

这会让数据修正、版权处理和错误分析几乎不可执行。可追溯性应从第一天设计。

一份上线前检查表

  • 每个来源都有快照、许可证和责任人；
  • 清洗规则有分域精确率与误杀率；
  • 精确、段落和近似去重均有统计；
  • PII、凭据和高风险内容有分级策略；
  • 所有评测集有去污染报告；
  • 数据配比由 token 统计和代理实验支持；
  • Tokenizer 在各语言和领域上做过压缩率分析；
  • packing 不跨样本泄漏，EOS 与 mask 行为经过单元测试；
  • shard 有校验和，样本能回溯原文；
  • 训练清单、配置和代码版本被冻结。

总结

预训练数据工程的目标不是制造一个最大的文本目录，而是制造一个可解释、可复现、可追溯的学习分布。

真正可靠的配方会同时回答四个问题：模型看到了什么，为什么让它看到这些，哪些内容被排除了，以及这些决定怎样改变了能力与风险。把这四件事说清楚，训练才真正有了可控的起点。
