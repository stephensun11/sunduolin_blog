MoE 模型原理及应用：路由、负载均衡与专家并行

2026-07-13 AiTech LLM 基础技术 LLM MoE Sparse Model Router Expert Parallelism Mixtral

Mixture of Experts（MoE）经常被概括为“用更少计算激活更多参数”。这句话抓住了稀疏性的直觉，却没有说明代价：参数仍要存储和通信，路由会造成不均衡，专家并行引入 all-to-all，训练与服务的性能也取决于 token 分布。

本文聚焦 Transformer 中最常见的稀疏前馈 MoE。它通常替换部分 dense FFN，而 attention 仍由所有 token 共享。

从 dense FFN 开始

一个简化的 Transformer 前馈层可写为：

\[ \operatorname{FFN}(x)=W_2\,\sigma(W_1x) \]

若使用 SwiGLU，则有两条上投影：

\[ \operatorname{SwiGLU}(x)=W_{\text{down}} \left(\operatorname{SiLU}(W_{\text{gate}}x)\odot W_{\text{up}}x\right) \]

每个 token 都经过同一组参数。扩大 FFN 宽度会同时增加参数量和每 token FLOPs。

MoE 准备 (E) 个专家 FFN，只为每个 token 选择少量专家。总参数量可以大幅增加，而每 token 激活参数量取决于 top-k。

Router 怎样选择专家

对 token 表示 (x)，router 产生 logits：

\[ z=W_rx \]

再得到门控概率：

\[ p_i(x)=\frac{\exp(z_i)}{\sum_{j=1}^{E}\exp(z_j)} \]

选出 top-k 专家集合 (S(x)) 后，输出可写成：

\[ y=\sum_{i\in S(x)}\tilde p_i(x)\,E_i(x) \]

其中 (\tilde p_i) 可以是原概率，也可以在选中专家间重新归一化。不同实现对权重、归一化、共享专家和残差路径的定义不同。

Top-1 与 Top-2

Top-1 每 token 激活一个专家，计算和通信较低；Top-2 激活两个专家，通常提供更多容量和路由冗余，但成本更高。

Switch Transformer 以简化的 top-1 routing 展示了大规模稀疏训练；Mixtral 则在每层专家中为每个 token 选择两个。

“总参数”与“每 token 激活参数”必须同时报告。MoE 模型的总参数量不能直接与 dense 模型参数量按相同 FLOPs 解释。

一个 top-2 路由的手算例子

假设有 4 个专家，某个 token 的 router logits 是：

expert 0:  2.1
expert 1:  0.4
expert 2:  1.3
expert 3: -0.2

对 4 个 logits 做 softmax，概率近似为：

[0.577, 0.105, 0.259, 0.058]

Top-2 选择 expert 0 和 expert 2。若在选中专家之间重新归一化：

\[ \tilde p_0=\frac{0.577}{0.577+0.259}\approx0.690 \]
\[ \tilde p_2=\frac{0.259}{0.577+0.259}\approx0.310 \]

最终输出为：

\[ y\approx0.690E_0(x)+0.310E_2(x) \]

下一个 token 会重新计算 router，可能选择完全不同的专家。MoE 的稀疏性是 token-level 的动态选择，不是整段文本永久绑定某个专家。

Router 学到的是什么

router 与主模型一起通过最终语言模型 loss 训练。只要某个路由决定让输出更接近正确 token，梯度就会调整 router 和被选专家。

Top-k 是离散选择，未选专家通常得不到该 token 的主任务梯度；选中专家和 gate 权重仍可对 loss 求导。不同实现还会加入噪声、辅助损失或无辅助损失的偏置更新来维持负载。

一次 MoE 前向的完整路径

设输入隐藏状态为 [B,T,d]，展平后有 (M=B\times T) 个 token：

1. router:   [M, d] -> [M, E]
2. top-k:    每个 token 得到 k 个 expert id 和 gate weight
3. dispatch: 按 expert id 对 token 排序、分桶
4. exchange: token 发往专家所在设备（expert parallel 时）
5. experts:  每个专家对自己的 token 做 FFN
6. exchange: 专家输出发回 token 原设备
7. combine:  按 gate weight 加权并恢复 [B,T,d]

简化伪代码如下：

def sparse_moe(x, router, experts, top_k=2):
    shape = x.shape
    flat = x.reshape(-1, shape[-1])             # [M, D]

    router_logits = router(flat)                # [M, E]
    probs = router_logits.softmax(dim=-1)
    weights, expert_ids = probs.topk(top_k, dim=-1)
    weights = weights / weights.sum(-1, keepdim=True)

    output = torch.zeros_like(flat)
    for expert_id, expert in enumerate(experts):
        token_idx, slot_idx = torch.where(expert_ids == expert_id)
        if token_idx.numel() == 0:
            continue
        expert_out = expert(flat[token_idx])
        output.index_add_(
            0,
            token_idx,
            weights[token_idx, slot_idx, None] * expert_out,
        )

    return output.reshape(shape), router_logits

这段代码便于理解，但真实 GPU 实现不会逐专家 Python 循环。它会使用 token permutation、grouped GEMM、fused kernel 和 all-to-all，以减少小 kernel 与随机访存。

为什么只把 FFN 换成专家

Transformer 中 FFN 参数通常占很大比例，而且每个 token 的 FFN 计算彼此独立，适合先路由再批量矩阵乘。

Attention 则负责 token 间交互，若每个 token 使用不同 attention 参数，KV cache、批处理和通信会复杂得多。因此主流 sparse MoE 多保留共享 attention，只把部分或全部 FFN 替换成专家。

这不是硬性规则，而是参数容量、计算规则性与系统复杂度之间的实用折中。

Capacity：专家一次能收多少 token

一个 batch 有 (T) 个 token，(E) 个专家，top-k 为 (k)。平均每个专家接收约 (kT/E) 个 token。工程上会设置 capacity factor：

\[ C=\left\lceil \text{capacity\_factor}\times\frac{kT}{E}\right\rceil \]

如果某个专家被分配超过 (C) 个 token，系统必须决定：丢弃超额 token、转给次选专家、使用更大的动态 buffer，或做其他路由修正。

capacity factor 太小会丢 token 或降低质量；太大则浪费显存和计算，并让 straggler 更严重。这个参数与 batch token 数和路由分布有关，不能脱离运行规模设置。

为什么会负载不均衡

如果 router 发现某些专家在早期略占优势，更多 token 会流向它们，热门专家得到更多梯度，可能形成正反馈；其他专家则训练不足。这会同时损害模型容量和硬件利用率。

一种常见辅助损失会同时考虑“被分配比例”和“平均路由概率”。设：

  • (f_i) 为 batch 中送给专家 (i) 的 token 比例；
  • (P_i) 为专家 (i) 的平均路由概率。

可构造：

\[ \mathcal{L}_{\text{balance}}=E\sum_{i=1}^{E} f_iP_i \]

总目标为语言模型损失加权辅助损失。系数过小无法平衡，过大则可能迫使均匀路由，牺牲有意义的专家分工。

还常见 router z-loss，用于抑制 router logits 过大、改善数值稳定性。具体公式和归一化要以实现为准。

Token-choice 与 Expert-choice

最常见的 token-choice routing 是“每个 token 选 top-k 专家”。它保证每个 token 获得固定计算量，但不能保证专家负载一致。

Expert Choice Routing 反过来让每个专家选择固定数量的 token，从而天然控制容量，但一个 token 可能被零个、一个或多个专家选择，训练和自回归服务的语义与实现会更复杂。

路由策略没有脱离场景的绝对优劣。训练吞吐、推理确定性、批量大小和目标硬件都会影响选择。

Shared Expert 在解决什么

如果所有知识都通过路由专家处理，通用模式会在多个专家中重复学习。共享专家让每个 token 都经过一条 dense 路径，路由专家则学习增量和特化能力。

DeepSeekMoE 提出更细粒度专家和 shared expert isolation，目标是提升专家专业化并减少知识冗余。

共享专家增加每 token 固定计算，因此需要和路由专家数量、宽度共同核算。它也不保证专家会自动按人类可解释领域分工。

MoE 的训练通信

专家通常分布在不同设备上。一次 MoE 层包含：

本地 token 表示
  -> router 与分桶
  -> all-to-all：token 发往专家所在设备
  -> 专家 FFN 计算
  -> all-to-all：结果返回原设备
  -> 按门控权重聚合

这就是 expert parallelism。与 tensor parallelism 在每个矩阵内切分不同，expert parallelism 主要按专家切分参数和 token。

通信量与 token 数、hidden size、top-k 和数据类型有关。即使理论 FLOPs 很低，跨节点 all-to-all 也可能成为瓶颈。专家放置应尽量利用节点内高速互联，并避免让路由频繁跨慢速网络。

与其他并行方式怎样组合

大规模 MoE 常组合：

  • data parallel：复制模型处理不同 batch；
  • tensor parallel：切分 attention 或单个大专家；
  • pipeline parallel：按层切 stage；
  • expert parallel：把不同专家放到不同 rank；
  • sequence/context parallel：切分长序列激活。

并行维度乘积必须匹配 world size。更重要的是，各维度的通信发生在不同位置，不能只根据 GPU 数拼出一个可整除配置。应基于拓扑测量 all-reduce、all-to-all 和 point-to-point 的竞争。

为什么 MoE 训练更容易出现数值问题

路由使每个专家看到的 token 数和分布不断变化。常见风险包括：

  • 某专家长时间几乎没有梯度；
  • 小 batch 下路由统计方差大；
  • router logits 过大导致概率饱和；
  • 热门专家出现更大梯度或 overflow；
  • capacity 溢出导致 token 丢弃；
  • 混合精度下门控排序对微小误差敏感。

监控至少包括每专家 token 数、门控概率、溢出/丢 token 比例、辅助损失、专家梯度范数和 all-to-all 时间。只看总 loss 很难发现一个专家已经“死亡”。

MoE 的推理不是天然更快

自回归 decode 每个 step 的 token 数可能很少。此时专家 kernel 太小，难以充分利用 GPU；动态路由又使连续批处理和缓存管理更复杂。

推理成本至少包含：

  • 所有专家权重的存储或跨设备访问；
  • router 计算和 token 排序；
  • 小 GEMM 或 grouped GEMM；
  • expert parallel 的 all-to-all；
  • 热门专家造成的负载倾斜；
  • 多副本部署时的专家复制成本。

因此 MoE 的优势通常是“在相近激活 FLOPs 下获得更大参数容量”，不是“任何硬件上延迟都更低”。小 batch、低延迟场景尤其需要实测。

MoE 模型的典型应用

1. 扩大预训练模型容量

MoE 最直接的用途是在控制每 token 计算量的同时增加总参数。更多参数可以容纳更丰富的模式，而 router 只激活其中一小部分。

这适合训练预算受每 token FLOPs 限制、但集群内存和高速互联足以容纳专家的场景。收益并非免费：总权重 checkpoint、更复杂的通信和训练稳定性都会增加成本。

2. 多语言和多领域建模

不同语言、代码、数学和自然语言可能形成不同路由偏好。Switch Transformer 在其多语言实验中报告了相对 mT5 基线的改善。

但 router 不接受“语言专家”标签监督。专家可能按语法、token 位置、抽象特征或训练阶段分工，不能看到路由热力图后就给专家随意命名。

3. 通用基础模型和聊天模型

MoE 可用于 base model 的预训练，也可以继续做 SFT 和偏好对齐。对齐阶段仍然只有被路由到的专家接收特定 token 的主任务梯度，因此需要监控专家利用率和通用能力回归。

4. 多模态模型

在文本、图像、音频等 token 共同进入 Transformer 时，MoE 可以增加共享主干的条件计算容量。专家可能形成模态偏好，也可能学习跨模态特征。

如果设计目标是硬性隔离模态，显式 modality router 或独立 tower 更可控；普通 learned router 不保证自动得到期望分工。

5. 资源受限的在线推理

MoE 有时能以较少激活参数达到较大 dense 模型的质量，但它不天然适合单 GPU 边缘设备。全部专家仍需存储，跨 GPU 专家还会产生通信。

在线服务更适合具备以下条件的场景：

  • 多 GPU/多节点能常驻全部专家；
  • batch 足够大，可形成有效 grouped GEMM；
  • 网络 all-to-all 性能好；
  • 推理引擎原生支持 expert parallel 和连续批处理。

代表模型怎样使用 MoE

Switch Transformer：Top-1 简化路由

Switch Transformer 每个 token 只选择一个专家，减少计算和通信复杂度。论文展示了稀疏模型可扩展到万亿参数，并系统讨论了 capacity、辅助负载均衡和低精度稳定性。

它的重要性不只是模型规模，而是把“如何让稀疏专家可训练”变成一套相对清晰的工程配方。

Mixtral 8x7B：每层 8 个专家、激活 2 个

Mixtral 在每个 MoE 层设置 8 个 FFN 专家，每个 token 选择 2 个。论文报告模型总共可访问约 47B 参数，推理时每 token 激活约 13B 参数。

“8x7B”不是 8 个完整 7B Transformer：attention、embedding 和其他共享部分不会复制 8 份，因此不能直接算成 56B。

DeepSeekMoE：细粒度专家与共享专家

DeepSeekMoE 把大专家拆成更细粒度的 routed experts，并加入 shared experts。共享专家承载常见模式，路由专家提供条件容量，目标是减少不同专家重复学习通用知识。

DeepSeek-V3：大规模 MoE 系统案例

DeepSeek-V3 报告 671B 总参数、每 token 激活 37B 参数，采用 DeepSeekMoE 架构。它还使用辅助损失自由的负载均衡策略，避免过强 auxiliary loss 直接干扰主任务目标。

“无辅助损失”不等于“不做负载均衡”，而是把平衡控制从加到语言模型目标上的辅助项，改成其他路由偏置机制。

MoE 与“多模型集成”有什么区别

传统 ensemble 让多个完整模型都处理输入，再平均或投票；MoE 通常是一个端到端模型，专家只是层内组件。

维度 MoE Ensemble
路由粒度 token / layer 请求或完整样本
共享部分 attention、embedding 等大量共享 多个完整模型通常独立
训练 常端到端联合训练 可独立训练后组合
计算 只激活 top-k 专家 常运行多个完整模型
输出组合 层内隐藏状态加权 最终预测加权/投票

MoE 也不同于产品层面的“模型路由”：后者可能按请求选择代码模型、视觉模型或便宜模型，不发生 token 级专家计算。

什么时候应该选择 MoE

适合优先评估：

  • 希望扩大模型容量，但不能按比例增加每 token FLOPs；
  • 有高速 GPU 互联和成熟分布式训练基础设施；
  • 训练 batch/token 数足够大，能摊薄路由与通信；
  • 推理侧能够常驻所有专家并支持 grouped GEMM；
  • 团队能长期监控路由、负载和专家退化。

更适合 dense 模型：

  • 单机或边缘部署，显存首先受总权重限制；
  • 极低并发、延迟优先；
  • 网络 all-to-all 较弱；
  • 训练规模不大，稀疏路由统计不稳定；
  • 工程团队需要简单、可移植的模型格式。

参数量和显存怎样计算

设每层 dense attention 参数为 (P_a)，单个专家 FFN 参数为 (P_e)，有 (E) 个专家，top-k 为 (k)。

总参数近似：

\[ P_{\text{total}}\approx P_a+E P_e+P_{\text{router}} \]

每 token 激活参数近似：

\[ P_{\text{active}}\approx P_a+kP_e+P_{\text{router}} \]

但显存仍更接近总参数，因为未激活专家的权重也要驻留某处。若权重从 CPU 或远端动态换入，延迟通常会非常高。

微调 MoE 的特殊问题

全量微调要更新所有专家，但某个小数据集可能只激活少数专家，造成能力偏移。LoRA 微调也需要决定：

  • 只适配共享 attention；
  • 适配所有专家；
  • 只适配被选中的专家；
  • 是否训练 router；
  • 是否保留负载均衡损失。

冻结 router 可以保持原路由分布，但可能限制领域适配；训练 router 则可能导致专家塌缩。需要同时比较任务效果和路由统计。

一个正确的性能比较

比较 dense 与 MoE 时至少对齐：

  • 训练 token 数和数据配方；
  • 每 token 前向/训练 FLOPs 的估算口径；
  • 总参数与激活参数；
  • 上下文长度和 batch token 数；
  • GPU 型号、互联和并行策略；
  • 质量指标、吞吐、延迟和峰值显存。

只说“8x7B 因此相当于 56B”并不严谨。专家参数、共享模块、top-k 和模型具体维度都会改变总参数与激活计算。

常见误区

每个专家会自动成为一个明确领域专家

模型可能形成语法、位置或抽象特征上的分工，未必对应“数学专家”“代码专家”这类人类标签。

未选中的专家不占任何成本

它们不参与该 token 的 FFN 计算，但仍占权重存储、checkpoint、加载和分布式管理成本。

top-1 一定比 top-2 快一倍

总延迟还受通信、kernel 利用率和负载影响，速度不会简单按 k 线性缩放。

负载越均匀越好

完全均匀可能抹掉有价值的专业化。目标是避免硬件和学习塌缩，而不是禁止路由偏好。

实施检查表

  • 明确报告总参数、激活参数、top-k 和专家数；
  • router 与门控归一化有单元测试；
  • capacity overflow 行为明确且可监控；
  • 每专家 token、概率和梯度有可视化；
  • all-to-all 在目标拓扑上做过基准测试；
  • checkpoint 能正确恢复专家、router 和 optimizer 分片；
  • 微调前后比较路由分布和通用能力；
  • 服务压测包含真实 batch、输入/输出长度和热点请求分布。

一个 MoE 训练监控面板

除了普通 loss、梯度和吞吐，至少记录：

指标 异常意味着什么
每专家 token 数 热门专家、死亡专家或数据偏斜
top-k 概率与熵 router 过度自信或接近随机
capacity overflow token 被丢弃/改路由的风险
每专家梯度范数 专家学习速度差异与数值异常
router z-loss / balance loss 路由稳定性与平衡代价
all-to-all 时间 网络拓扑或负载倾斜瓶颈
grouped GEMM 利用率 每专家 token 太少、kernel 过碎
分域路由分布 某语言或领域是否集中到少数专家

路由图只能帮助诊断，不能直接证明专家具备某种语义能力。还要通过 expert ablation、替换或受控数据实验验证因果作用。

总结

MoE 把模型扩展问题从“所有 token 经过所有参数”改成“router 为 token 选择少量参数”。它用稀疏激活换取更大的总容量，但也把难题转移到了路由、容量、负载和通信上。

理解 MoE 时，始终把三本账分开：总参数决定存储规模，激活参数影响理论计算，实际 token 路由和网络拓扑决定系统性能。三者缺一，任何“更大但更省”的结论都不完整。
